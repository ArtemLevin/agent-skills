# Архитектура обвязки

## Стабильные границы расширения 1.0

Интеграции провайдеров реализуют capability и execution contracts из `agentkit.model_runtime`. Стабильные фазы: `plan`, `implementation`, `review` и `targeted_fix`. Прямые API-провайдеры могут обслуживать read-only фазы; изменение рабочего дерева разрешено только локальному исполнителю с явно заявленной capability.

Ресурсы проекта устанавливаются через версионированный manifest. Новое расширение должно определить стратегию ресурса, версионированную JSON Schema, детерминированную сериализацию, CLI/Make parity и поведение при недоступном optional provider.

## 1. Почему один управляющий агент

Несколько постоянно активных агентов увеличивают число повторных чтений, конфликтующих рекомендаций и model round trips. В этой архитектуре один оркестратор хранит состояние задачи, а узкие навыки подключаются как временные режимы мышления.

Навык — не персона и не отдельный чат. Это компактный контракт:

- когда активироваться;
- какие входные данные требовать;
- какие решения принимать;
- какой структурированный результат возвращать;
- когда остановиться.

## 2. Поток данных

```text
User request
  │
  ▼
Task packet ──► Triage result ──► Context ledger
  │                                  │
  └──────────────► Requirements contract
                                     │
                                     ▼
                              Approved change plan
                                     │
                                     ▼
                                  Patch
                                     │
                         ┌───────────┴───────────┐
                         ▼                       ▼
                 Verification report      Review findings
                         └───────────┬───────────┘
                                     ▼
                              Completion report
```

Каждый этап получает дельту, а не всю историю. Например, ревьюеру передаются контракт, diff, результаты проверок и подтверждённые инварианты.

## 3. Progressive disclosure

На старте оркестратор знает только имя и описание навыка. Полный `SKILL.md` загружается после выбора в `task-triage`. Дополнительные справочники и скрипты читаются только по прямой необходимости.

Это уменьшает системный prompt и препятствует взаимному загрязнению инструкций.

## 4. Контракты этапов

### Triage result

Определяет тип задачи, режим, риск, предполагаемую область изменений, обязательные и необязательные навыки, а также лимиты чтения и итераций.

### Context ledger

Хранит только подтверждённые факты:

- точки входа и владельцев инвариантов;
- непосредственные зависимости;
- существующие тесты;
- просмотренные символы;
- неизвестные и отвергнутые предположения.

### Requirements contract

Отделяет acceptance criteria от non-goals, фиксирует совместимость и запрещённые изменения.

### Change plan

Привязывает каждый шаг к файлу или символу, причине, проверке и условию отката.

### Review report

Содержит только проверяемые находки с тяжестью, доказательством и минимальным исправлением.

## 5. Контрольные шлюзы

1. **Scope gate** — изменение соответствует контракту и approved scope.
2. **Implementation gate** — diff минимален и следует стилю проекта.
3. **Verification gate** — выполнены проверки, соответствующие риску.
4. **Review gate** — отсутствуют P0/P1.
5. **Completion gate** — результат и остаточные риски зафиксированы.

## 6. Модель токенов

Основные источники экономии:

- выбор навыков по событию, а не постоянная загрузка;
- symbol-first поиск;
- ограничение количества первоначально читаемых файлов;
- context ledger вместо повторного чтения;
- передача diff summary между этапами;
- narrow-first запуск тестов;
- ограничение числа циклов implementation-review;
- запрет на P3-полировку после прохождения completion gate.

## 7. Расширение

Специализированный навык стоит добавлять, когда:

- возникает отдельный класс рисков;
- для него существуют уникальные проверки;
- общий ревьюер регулярно пропускает ошибки этого класса;
- навык можно активировать по однозначным триггерам.

Не стоит создавать навык для отдельного фреймворка, если различия сводятся к нескольким командам; такие сведения лучше хранить в `references/` существующего навыка.
